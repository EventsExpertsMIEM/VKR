Events Service Page

Сервис мероприятий конгрессной деятельности

Это страница репозитория с кодом сервера Web-сервиса для организации и поддержки конгрессной деятельности - для автоматизации и упрощения взаимодействия "участник-организатор-исполнитель".

Основные функции сервиса:

  • Отслеживание мероприятий
  • Создание мероприятий и их настройка
  • Отслеживание статуса подготовки мероприятий и модерацией участников
  • Регистрация на мероприятия и получение информационной поддержки

Repo

Как работать в репозитории:

  • Создавать под фичи и задачи отдельные ветки
  • После завершения работы создавать ПР с указанием меня для проверки
  • ПР просьба оформлять грамотно
  • ПР делаются в ветку master

Структура репозитория:

  • В app лежит код самого бекенда
    • api - REST API
    • db - структура БД и инициализация
    • logic - внутрення логика
    • static - статические файлы для страниц
    • templates - шаблоны страниц для jinja2
    • web - urlы
    • __init__ - инициализация flask и модулей
    • congig - конфиг для всего сервера
    • errors - обработка ошибок
    • logging_config - конфиг логгеров сервера
    • mails - почта
  • main - запуск сервера
  • requirements.txt - необходимые pip пакеты
  • setup.sh - шаблон файла инициализации конфига в переменные окружения (shell)
  • setup.bat - шаблон файла инициализации конфига в переменные окружения (win)

What we use

Основное:

  • Язык программирования сервера - Python3
  • СУБД - PostgreSQL
  • ORM для БД - SQLAlchemy
  • Web-framework - Flask
  • Web-интерфейс - Bootstrap4

pip модули:

  • Хэширование паролей на сервере - bcrypt
  • Авторизация - Flask-Login
  • Работа с эл. почтой - Flask-Mail
  • WSGI coroutines - gevent
  • UID генерация - nanoid
  • Подключение к БД (PostgreSQL) - pg8000 и psycopg2-binary
  • Управление реквестами - requests

DB setup

Почему PostgreSQL:

  • Хорошая докумантция и большое сообщество.
  • Поддержка множества дополнительных типов, например UUID, JSONB и другие.
  • Следует ACID (в отличии от MySQL/InnoDB).
  • Шардирование из коробки.

Можно поднять PostgreSQL в облаке (например Heroku) или же развернуть на своих ресурсах по инструкции ниже:

1) Ставим пакет:

  • Ubuntu:
    • sudo apt install -y postgresql
  • CentOS 7:
    • sudo yum install postgresql-server postgresql-contrib
    sudo postgresql-setup initdb
    sudo systemctl start postgresql
  • Windows:
    • Скачать с сайта и установить.

2) Переходим в суперюзера бд:

  • Linux:
    • sudo su - postgres
  • Windows:
    • Залогиниться через консоль в админа постгреса.

3) Создаём новую роль:

createuser %usename% --pwprompt

4) Создаём саму БД:

createdb -O %username% %dbname%

5) Подключаемся к новой БД и даем доступы:

psql %dbname% 
GRANT ALL ON DATABASE %dbname% to %username%;

6) CENTOS ONLY! Отредактировать конфиг-файл:

cd /var/lib/pgsql/data/
sudo nano/vim ph_hba.conf
Заменить METHOD на md5 у local, ipv4, ipv6 connections (3 записи).

Nginx


server {

    # Прокси сервер открывается на стандортном порту для HTTP трафика, но для дополнительной безопасности
    # включено обязательное шифрование с помощью директивы ssl
    listen 80 ssl;

    # В качестве переменной server_name необходимо установить домен, на котором будет работать приложение
    server_name congress-events.ru;

    # Пути, по которым находятся TLS сертификат и приватный ключ от него
    # - www.congress-events.ru.crt - Файл сертификата
    # - www.congress-events.ru.key - Приватный ключ сертификата
    # В качестве примера приведены стандартные пути размещения для сертификатов и ключей в CentOS 7
    ssl_certificate /etc/pki/tls/certs/www.congress-events.ru.crt;
    ssl_certificate_key /etc/pki/tls/private/www.congress-events.ru.key;

    # При попытке подключится к HTTPS серверу посредством HTTP возникнет ошибка под номером 497
    # Данная директива указывает серверу на то, что при возникновении ошибки 497
    # нобходимо произвести переподключение с использованием протокола HTTPS
    error_page 497 301 =307 https://$http_host$uri;

    # Настройка проксирования запросов
    # Директива location / говорит о необходимости перенаправлять все запросы 
    location / {
        # Необходимо установить заголовок Host включающий в себя полный запрос,
        # так как по умолчанию в заголовке Host не учитывается номер порта, на который пришел запрос
        # что приводит к некорректному формированию редирктов
        proxy_set_header Host $http_host;

        # Директива proxy_pass устанавливает адрес, на который необходимо перенаправлять запросы
        proxy_pass https://192.168.255.99:45000/;
    }
}

Running

Сервис берёт свои настройки (БД, параметры почты и др.) через переменные окружения, поэтому рекомендуется использовать специальный скрипт (например setup.sh/setup.bat) и инициилизировать переменные окручения через . setup.sh / source setup.sh / setup.bat.

При первом запуске необходимо указать --create-tables, чтобы создать в БД нужные таблицы (или сбросить, если необходим перезапуск с очисткой)

Запуск происходит через python3 main.py.

REST API (need to complete)

Далее идет описание REST API.

Основным форматом является json (но не всегда), для чего необходимо указывать в заголовках application/json. В случае отсутсвия данного заголовка или же некорретно составленного тела будет возвращена ошибка с http кодом 415.

Ниже приведены описания всех возможных ответов сервера:

200 - основной ответ в случае успешного выполнения запроса.
201 - ответ при успешном создании.

400 - ответ при неправильных ключах json.
401 - ответ незалогиненому пользователю.
403 - ответ при отсутствии доступа к выполняемому действию.
404 - ответ при отсутствии запрашиваемого пути или запрашиваемых данных.
405 - ответ при отсутствии метода у данного пути.
409 - ответ при конфликте состояния.
415 - ответ при некорректном json.
422 - ответ при некорретных поданых данных.

Reports

Ручки, связанные с докладами.

Все ручки данной категории имеют префикс /event (т.е. первая ручка / является /event/).

  • Загрузка доклада на мероприятие
  • Принимает
    • [id] - id мероприятия
    • multipart-form-data
      • file - файл с докладом (pdf, word/x, ppt/x)
  • 200 OK
    • description
  • 401 - не залогинен

  • 404 - мероприятие не найдено
